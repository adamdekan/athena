# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What ATHENA is

ATHENA is a fully offline, privacy-first voice assistant (no cloud, telemetry, or API keys; zero runtime Python) that chains speech-to-text → LLM → text-to-speech entirely in C++ on a single consumer GPU plus large system RAM. The pipeline:

- **STT + LLM**: `whisper-talk-llama` (a patched whisper.cpp `talk-llama`) captures the mic (SDL2), transcribes with Whisper `small.en`, and runs the **Qwen3.5-397B-A17B** MoE chat model *in-process* (its routed experts offloaded to system RAM via `--cpu-moe`). It also does Silero VAD, prosodic endpointing, emotion2vec emotion tagging, barge-in, and cross-session memory.
- **TTS**: `talk-llama` streams generated assistant sentences to a trigger file; the `orpheus-speak` daemon reads them, asks a local `llama-server` (hosting the **Orpheus-3B** GGUF) for SNAC audio tokens over HTTP/SSE, decodes them to PCM with the **SNAC 24 kHz ONNX** vocoder, and plays via `aplay`.

**Three processes run concurrently**, all started by `launch-athena-397b.sh`: `llama-server` (Orpheus TTS backend, port 8080), `orpheus-speak` (SNAC decoder/player), and `whisper-talk-llama` (STT+LLM, run in the foreground/blocking). `speak-daemon.sh` is **not** a launched process — it is only registered as `talk-llama`'s `-s` speak handler and stays dormant because the active path is `--stream-file` (see gotchas).

> Two LLMs run at once. `llama-server` on :8080 serves **only** Orpheus TTS. The conversational Qwen model runs **inside** `whisper-talk-llama`, not over HTTP.

## Model file layout note

All model files live **flat in `models/`**, including the five Qwen shards (`Qwen3.5-397B-A17B-UD-Q3_K_XL-0000{1..5}-of-00005.gguf`). `launch-athena-397b.sh:29` previously pointed `QWEN_MODEL` at a nonexistent `models/QWEN-3.5-397B/` subdirectory, which made the preflight loop (`launch-athena-397b.sh:636-638`) abort every launch (see the failed run in `athena.log` at 2026-06-30 14:44:26). That path has been corrected to `models/Qwen3.5-...-00001-of-00005.gguf`; if you relocate the shards into a subdir, update `QWEN_MODEL` to match.

## Build commands

ATHENA actually depends on exactly two compiled binaries (`llama-server`, `whisper-talk-llama`) plus its own `orpheus-speak`. All the other `llama-*`/`whisper-*`/test binaries the CMake builds produce are unused byproducts.

**1. llama.cpp** (produces `llama.cpp/build/bin/llama-server`):
```bash
cd /home/user/ATHENA/llama.cpp
cmake -B build -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES="120a;86" \
  -DGGML_CUDA_FORCE_CUBLAS=OFF -DGGML_CUDA_FA_ALL_QUANTS=ON -DGGML_CUDA_F16=ON \
  -DGGML_SCHED_MAX_COPIES=1 -DGGML_NATIVE=ON -DGGML_LTO=ON -DCMAKE_BUILD_TYPE=Release \
  -DGGML_CUDA_GRAPHS=ON -DLLAMA_BUILD_UI=OFF
cmake --build build -j 12
```

**2. whisper.cpp with the custom talk-llama** (produces `whisper.cpp/build/bin/whisper-talk-llama`). The FIVE custom files MUST be copied in **before** configuring (see the `patches/talk-llama/` layout note above):
```bash
cd /home/user/ATHENA/whisper.cpp
cp ../patches/talk-llama/* examples/talk-llama/
cmake -B build -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES="120a;86" \
  -DONNXRUNTIME_ROOT=/home/user/ATHENA/onnxruntime-linux-x64-gpu_cuda12-1.27.0 \
  -DGGML_CUDA_FORCE_CUBLAS=OFF -DGGML_CUDA_FA_ALL_QUANTS=ON -DGGML_CUDA_F16=ON \
  -DGGML_SCHED_MAX_COPIES=1 -DGGML_NATIVE=ON -DGGML_LTO=ON -DWHISPER_SDL2=ON \
  -DGGML_CUDA_GRAPHS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build -j 12
```
`-DONNXRUNTIME_ROOT` enables emotion2vec (defines `ATHENA_EMOTION_ORT=1`); `-DWHISPER_SDL2=ON` enables mic capture. `whisper-talk-llama` bundles its **own** full copy of the llama sources (`examples/talk-llama/llama*.cpp`) — it does not link `libllama` or depend on the `llama.cpp/build` tree.

**3. orpheus-speak** (produces `orpheus/build/orpheus-speak`):
```bash
cd /home/user/ATHENA/orpheus
cmake -B build -DONNXRUNTIME_ROOT=/home/user/ATHENA/onnxruntime-linux-x64-gpu_cuda12-1.27.0 \
  -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)
```
`-DONNXRUNTIME_ROOT` is mandatory (`CMakeLists.txt` FATAL_ERRORs without it). The ORT `.so` is resolved at runtime via a RUNPATH baked into the binary — moving/renaming the onnxruntime dir breaks the binary unless rebuilt. `cmake --install` is broken (the `install()` rule references a nonexistent `speak.sh`); the project is always run in-place from `build/`.

ONNX Runtime is **never built** — it is a prebuilt tarball extracted in place.

## Running ATHENA

```bash
./launch-athena-397b.sh                          # production launch (all defaults)
ATHENA_EMOTION_CALIBRATION=1 ./launch-athena-397b.sh   # per-class emotion calibration
./launch-athena-397b.sh 2>&1 | ts '[%Y-%m-%d %H:%M:%.S]' | tee athena.log   # timestamped log (needs moreutils)
```
The launcher takes **no positional args/subcommands**; every "mode" is an env var read near the top. It uses `sudo` heavily (nvidia-smi clocks/power/persistence, governor/EPP/C-states/THP/NUMA, stopping power-profiles-daemon) and will block on a password prompt if run non-interactively. All tuning is reverted by `cleanup()` on exit.

The **desktop icon path differs materially** from a direct shell run: the `Athena 397B.desktop` Exec line is the **only** place `GGML_CUDA_NO_PINNED=1` and `ATHENA_MPS=1` are set. A direct `./launch-athena-397b.sh` gets neither (no MPS, pinned memory on).

Inspecting individual pieces: the binaries can be run standalone, e.g. `orpheus/build/orpheus-speak --watch <trigger> --snac <snac.onnx> --play "aplay -q" --stream-tts -v`, or single-shot `orpheus-speak [opts] "text to speak"` (→ `/dev/shm/orpheus_tts.wav`).

## Architecture (how the launcher wires it together)

Main flow (`launch-athena-397b.sh`, bottom of file): `setup_gpu_performance` → `start_mps` (must precede any GPU proc so all inherit `CUDA_MPS_*`) → `setup_system_performance` → `setup_audio` → preflight `check_file` for all binaries + models (incl. all 5 Qwen shards) → start GPU telemetry sampler → launch `llama-server` and poll `/health` → launch `orpheus-speak` → launch `whisper-talk-llama` in the **foreground** (blocking). `trap cleanup EXIT INT TERM` + `set -e -E` with an ERR trap.

**Process 1 — Orpheus TTS server** (PID_LLAMA):
```
stdbuf -oL -eL llama-server -m <orpheus GGUF> \
  -c 13824 -np 2 -ngl 99 --host 127.0.0.1 --port 8080 \
  --cache-type-k f16 --cache-type-v f16 --cache-ram 0 -fa on -t 0
```
Readiness: `curl -s http://127.0.0.1:8080/health` (polled up to ~60 s; failure aborts launch). `-np 2` + continuous batching give parallel TTS slots (was `-np 4`; single-user needs no more, and it halves peak concurrent-decode / MPS stream count — CHANGES.MD §10). `GGML_CUDA_GRAPH_OPT` is **no longer set** (see gotchas).

**Process 2 — orpheus-speak** (PID_ORPHEUS):
```
orpheus-speak --watch <speak_tts.txt> --snac <snac24_dynamic_fp16.onnx> \
  --play "aplay -q" --stream-tts --diag -v --capture-dir <run_dir>/tts-capture
```
SNAC runs on CUDA device 0 (~1.7 GB VRAM; `--snac-cpu` forces CPU). `--diag` is omitted when `ATHENA_DIAG=0`; `--capture-dir` is omitted when `ATHENA_TTS_CAPTURE=0` (see *TTS garble-diagnosis capture* below).

**Process 3 — whisper-talk-llama** (foreground), pinned to cores 0-15:
```
taskset -c 0-15 whisper-talk-llama \
  -ml <Qwen shard 00001-of-00005> -mw <ggml-small.en.bin> \
  --ctx-size 80000 --mlock --no-mmap --cpu-moe -t 16 -ngl 99 -ctk bf16 -ctv bf16 -fa \
  --temp 0.7 --top-p 0.8 --top-k 20 --min-p 0.00 --presence-penalty 1.5 --repeat-penalty 1.0 \
  --reasoning off -s <speak-daemon.sh> -sf <speakfile.temp> --stream-file <speak_tts.txt> \
  --barge-in --barge-rms 0.0020 --barge-ratio 1.5 --barge-ms 150 --barge-blackout-ms 200 \
  <memory args> -p Igor -bn Athena -mt 256 -vms 25000 \
  --vad-engine silero --vad-model <ggml-silero-v6.2.0.bin> --vad-last-ms 400 --vad-window-ms 700 \
  --endpoint --endpoint-short-ms 800 --endpoint-long-ms 800 --endpoint-f0-fall 60 \
  --endpoint-extend-ms 1200 --endpoint-f0-rise 50 --endpoint-energy-rise 0.5 \
  --endpoint-energy-decay 4.0 --silero-min-run-ms 100 --silero-debug -pe
```
Dense/attention layers on GPU (`-ngl 99`); routed MoE experts on CPU and page-locked in host RAM (`--cpu-moe --no-mmap --mlock`, ~161 GiB resident). Qwen KV must be **bf16** — a quantized KV degraded output. Memory args are added only when `MEMORY_DIR` is set: `--memory <dir> --memory-words 2048 --time-refresh-min 15`, plus a `--personality-reflect-every 1` (`launch-athena-397b.sh:777`) — the latter is a **temporary demo edit** (lowers the personality-revision threshold to 10 so one session can fire a revision); drop the flag to revert to default behavior.

`GGML_CUDA_GRAPH_OPT` is **no longer set** on any process (CHANGES.MD §10 C1). It was previously an inline prefix on `llama-server` and `whisper-talk-llama`; it is an experimental multi-stream/buffer-reuse CUDA path with a known buffer-reuse race (the "unspecified launch failure" class) and no measurable benefit, so it was removed. Base CUDA graphs (`-DGGML_CUDA_GRAPHS=ON`) + kernel fusion stay on and self-disable at batch>1.

### Ports, sockets, and the TTS handoff protocol

The only TCP endpoint is `llama-server` on `127.0.0.1:8080` (`/health`, `/completion` SSE). `orpheus-speak`'s connection to :8080 is a **compiled-in default** (`--api http://127.0.0.1:8080/completion`), not passed by the launcher. The only other socket is the CUDA MPS control pipe `/tmp/nvidia-mps/control` (unrelated to TTS).

The TTS handoff is **file-based** (regular files + atomic `mv` + inotify + a polling `.done` sentinel) — not a named pipe or socket. Files (launcher paths, all under `/home/user/ATHENA/`):
- `speak_tts.txt` — trigger file. **Streaming path** (active): `talk-llama --stream-file` *appends* sentence lines ending with a `---END---` sentinel → inotify `IN_CLOSE_WRITE` → streaming session. **Legacy path**: `speak-daemon.sh` sanitizes markdown from `speakfile.temp` and atomically `mv`s it onto the trigger → inotify `IN_MOVED_TO` → batch mode (no `---END---`).
- `speak_tts.done` — written by orpheus-speak **after** audio fully drains (`COMPLETE`, or `INTERRUPTED <lines> <chars>`); then it deletes the trigger and flushes stale inotify events.
- `speak_tts.stop` — barge-in handoff: `talk-llama` creates it to abort playback; orpheus-speak kills audio within one ALSA period.
- `speakfile.temp` (legacy), `/dev/shm/orpheus_tts.wav` (WAV scratch).

### TTS garble-diagnosis capture

`orpheus-speak` has built-in (pure-C++, no Python) instrumentation for root-causing the intermittent TTS **garble**. `ATHENA_TTS_CAPTURE` (**default 1**) makes the launcher pass `--capture-dir <run_dir>/tts-capture`, dumping per TTS session: `.tokens` (raw model output), `.codes` (parsed audio codes, re-decodable), `.wav` (live decoded audio), `.meta`. Each session writes a few MB of WAV — set `ATHENA_TTS_CAPTURE=0` to disable. Two offline subcommands (each runs and exits):
```bash
# re-decode captured codes with a contention-free CPU reference
orpheus-speak --snac <snac24_dynamic_fp16.onnx> --snac-cpu --decode-codes <sess>.codes   # → <sess>.codes.redecode.wav
# correlation/RMS verdict between two WAVs (clean ≈0.999 MATCH; garble craters it → MISMATCH)
orpheus-speak --compare-wav <sess>.wav <sess>.codes.redecode.wav
```
MATCH ⇒ the codes/model are the fault; MISMATCH ⇒ the on-GPU SNAC decode corrupted the audio under contention. (The 20260630-230402 run's audio was 100% clean — no garble recurred under MPS.) The end-of-run crash there was a **GPU/GSP fault**, not garble — see the GSP gotcha below. A watchdog/auto-restart supervisor was designed but deferred (CHANGES.MD §10); `talk-llama` now at least fails **safe** on a GPU loss (brain-resilience, below) rather than aborting.

### GPU / model assumptions

Target hardware (baked into comments/values): ThinkPad P16 Gen 3, Intel Core Ultra 9 285HX (24 cores, no SMT), 192 GB DDR5-4000, single NVIDIA RTX PRO 5000 Blackwell Laptop 24 GB. VRAM is tight (~19 of 24 GB used). The 192 GB RAM requirement comes from the MoE experts living in host RAM under `--cpu-moe`. Qwen decode is bound by memory bandwidth (~6.4 tok/s), not GPU. Both CUDA builds target `CMAKE_CUDA_ARCHITECTURES=120a;86` (Blackwell sm_120a + Ampere).

### Diagnostics

`ATHENA_DIAG=1` (default) creates `athena-diag/<YYYYMMDD-HHMMSS>/` and spawns `athena-gpu-monitor.sh <run_dir> 1000` (1 Hz; §13a: each stream is now ONE persistent `nvidia-smi --loop-ms`, not a spawn-per-sample, to avoid UVM va-space churn) writing `gpu.csv`, `gpu-proc.log`, `gpu-throttle.log`, `xid.log`, `kmsg-full.log` (unfiltered kernel log — oops/panic backtraces, §11 F2), `power-state.log`; `llama-server` output → `orpheus-server.log`. A prior kernel panic is harvested into `athena-diag/crash-<ts>/` on the next launch. `ATHENA_DIAG=0` disables all of it (server output → `/dev/null`). Much of the GPU tuning and these knobs exist to chase a recurring NVIDIA **Xid 69** fault at the ~95 W power wall; `ATHENA_LOCK_MEMCLK`, `ATHENA_GPU_POWER_LIMIT`, `ATHENA_GPU_CLOCK_CAP`, `ATHENA_MPS` are off-by-default A/B levers. `demo-rec-start.sh`/`demo-rec-stop.sh` are an independent PulseAudio screen-recording tap, never invoked by the launcher.

## Gotchas (non-obvious)

- **Memory persistence requires a graceful spoken "Goodbye Athena".** That triggers a ~20-30 s Qwen memory-consolidation generation. **Ctrl+C does NOT write memory** (SIGTERM arrives first). Cleanup order is load-bearing: orpheus-speak and llama-server are killed first (5 s grace each), `talk-llama` **last** with a 60 s grace — a SIGKILL mid-decode under MPS (with ~161 GiB pinned host RAM) was observed to corrupt kernel page state.
- **`ATHENA_MPS=1` does not guarantee MPS actually engages.** `start_mps` (`launch-athena-397b.sh`) previously treated a merely *existing* `/tmp/nvidia-mps/control` pipe as "daemon already up" and returned without starting one — so a **stale** pipe left by a prior daemon that died without cleanup (a SIGKILLed `talk-llama`; the reuse path also "will NOT stop it on exit") made every subsequent launch silently run **without** MPS, the three GPU clients time-slicing one context. Both 2026-06-30 demo runs hit this despite the desktop icon setting `ATHENA_MPS=1`. Fixed (see `CHANGES.MD` §7): `start_mps` now liveness-probes the pipe (`get_default_active_thread_percentage | timeout 5 nvidia-cuda-mps-control`) and clears/rebuilds a stale one. **Diagnostic tell:** an **empty `mps-log/`** in the run's diag dir (and no `nvidia-cuda-mps-server` in `gpu-proc.log`) means MPS did **not** run — a daemon this launcher owns writes into `mps-log/`.
- **MPS on Debian needs `/usr/sbin` on PATH.** The MPS *server* (`nvidia-cuda-mps-server`) ships in `/usr/sbin`, which Debian (unlike Ubuntu) omits from the user PATH — so the control daemon can't exec it and every server dies `exited with status 1` (empty `mps-log/server.log`), leaving clients with `MPS client failed to connect ...`; llama-server then falls back to CPU while orpheus-speak's SNAC CUDA-EP `CreateSession` fails. Fixed (see `CHANGES.MD` §9): `start_mps` prepends `/usr/sbin` to PATH, then **self-checks** that a server actually spawns (`start_server -uid $(id -u)` + `get_server_list`) and falls back to no-MPS if not. orpheus-speak also now retries SNAC on the **CPU EP** if the CUDA session fails (prints the real ORT error) instead of exiting. **Diagnostic tell:** `Server N exited with status 1` in `mps-log/control.log` = the server can't launch.
- **The five `patches/talk-llama/` files must be copied into `examples/talk-llama/` before `cmake` configure** (`cp patches/talk-llama/* whisper.cpp/examples/talk-llama/`). An upstream sync or `git checkout .` erases them; the patched CMakeLists is required even on current master (upstream's own drops the emotion2vec wiring, and ATHENA's pre-DSA one fails to link there).
- **`speak-daemon.sh` is the dormant legacy batch path** (talk-llama's `--stream-file` bypasses the `-s` handler, so it never runs). Its `SPEAK_FILE`/`TRIGGER_FILE`/`DONE_FILE` vars must stay in sync with the launcher's (lines 87-89) — they previously omitted the `/ATHENA/` segment, which would have made the legacy `-s/-sf` path silently exit on a missing file; they have since been corrected to match, and its comment block refreshed from the old `QWEN-3.5-9B` example to the current 397B setup.
- **`orpheus-speak`'s compiled-in `--snac` default is `snac24_decoder.onnx`, which does not exist in the repo.** ATHENA works only because the launcher passes `--snac .../orpheus/snac24_dynamic_fp16.onnx` explicitly.
- `orpheus-speak` writes an optimized graph to `${snac_model}.optimized`; a GPU-optimized cache breaks a later CPU run, so the launcher `rm -f`s it before each start.
- **SSE scanning deliberately skips the final `"stop":true` event** — some llama-server builds repeat accumulated content there and would double the audio. Widen the guard if audio doubles at chunk end.
- **The launcher command line diverges from its own comments**: actual `--ctx-size 80000` (a comment argues for 131072) and `--temp 0.7 --top-p 0.8` (a comment cites temp=1.0/top_p=0.95). Trust the flags, not the prose.
- **README reconciled (2026-07-02, CHANGES.MD §17)**: the former doc/reality gaps (ORT 1.26 vs 1.27, CUDA 12.8 vs 12.9, gcc-12 vs GCC 14.2, generic `~/Orpheus` paths, missing Silero/driver/CUDA/stability sections) are fixed; the README now matches the CMakeCaches, launcher, and real layout, with live-verified download URLs (incl. the load-bearing `UD-Q3_K_XL/` subfolder in the Qwen resolve URLs and the debian12-repo quirk for CUDA 12.9/cuDNN). If code and README diverge again, trust the CMakeCache/launcher and update the README.
- **Version pins are README-documented but not verifiable from this checkout**: README claims llama.cpp release **b9253** and whisper.cpp commit **afa2ea544fb4b0448916b4a31ecd33c8685bd482** (the whisper.cpp commit is real in-tree; the llama.cpp tree has no `.git` and `build-info.cpp` reports build `0`/commit `unknown`). The Qwen3.5/GatedDeltaNet CUDA path is version-sensitive on Blackwell sm_120a, so do not blindly bump to upstream master.
- ONNX Runtime GPU requires the **full cuDNN 9.x** (not JIT, which adds runtime latency); ensure `/etc/ld.so.conf` includes the ORT lib path + `sudo ldconfig`, or the CUDA EP is unavailable.
- Qwen is a 5-shard split GGUF; the launcher passes only shard `00001-of-00005` to `-ml` and llama loads the rest, so a missing later shard fails minutes into load — preflight explicitly checks all 5.
- Barge-in / self-echo is **not fully tuned on open speakers** (Athena can interrupt herself); a headset (`ATHENA_AUDIO=direct`) is the reliable setup. A rolled-back interrupted turn deliberately keeps the spoken prefix with no end marker.
- SNAC token math: each 7-token Orpheus frame = 2048 samples = 85.3 ms (11.72 fps). Audio codes are `<custom_token_N>` with offset 10.
- **The Blackwell GSP firmware fault is the top stability risk, and MPS makes one fault kill all three processes.** A `NVRM: GspMsgQueueReceiveStatus: Bad checksum` (or GSP heartbeat/`GPU is lost`) in `xid.log`/`dmesg` is a known, unresolved open-driver 595.x + Blackwell bug (open-gpu-kernel-modules #1064/#1111) that corrupts the GSP↔driver RPC ring and takes the GPU context down. **GSP cannot be disabled on Blackwell.** Under CUDA MPS a single GPU the fault is reported to **every** client sharing it (one fault-containment domain — *by design*, no config isolates it) and the device is reset, poisoning all three ATHENA processes with a sticky CUDA error; the 20260630-230402 crash was exactly this (llama-server faulted → talk-llama SIGABRT'd 53 s later on its next GPU sync). MPS is still correct to keep (cooperative single-app; dropping it costs concurrency and would not prevent a device-level fault). Mitigation is **driver-side and staged, not applied**: run `apply-gsp-stability.sh` (installs `zzz-nvidia-athena-gsp-stability.conf` → disables Runtime-D3/S0ix so the GPU stays in D0, removing the power-state transitions #1064 ties to the fault; needs a reboot, zero compute cost). **⚠️ The `zzz-` prefix is load-bearing (CHANGES.MD §11 F1):** the distro's `/etc/modprobe.d/nvidia.conf` sets `NVreg_EnableS0ixPowerManagement=1`, and modprobe is C-locale-order **last-assignment-wins** — the original name sorted *before* `nvidia.conf` so S0ix stayed **enabled** and the 20260701-074343 run crashed (Xid 69 + kernel panic) with the mitigation silently defeated. `apply-gsp-stability.sh` now prints a precedence report; verify after reboot with `grep -i EnableS0ix /proc/driver/nvidia/params` (expect `0` — use `/proc`, not `/sys/module`, which is virtualized away in the GPU-passthrough container). **Diagnostic tell:** correlate `xid.log` GSP/Xid lines with `mps-log/server.log` "encountered a fatal GPU error" / "devices will be reset".
- **Capturing a kernel panic (CHANGES.MD §11 F2).** The 20260701 panic left no backtrace: the old `dmesg -w | grep "NVRM|Xid|nvidia"` follow discarded the oops body and never `fsync`'d. Now `athena-gpu-monitor.sh` runs **`athena-kmsg-logger`** (C++, `sudo -n`) which writes the **unfiltered** kernel log to `kmsg-full.log` + the NVRM/Xid/GSP subset to `xid.log`, `fdatasync`'ing every line. Run `apply-crash-capture.sh` once to build it and arm efi-pstore (`panic_on_oops=1`/`hardlockup_panic=1`/`panic=30`) + `NVreg_EnableGpuFirmwareLogs=1`; after a panic the next launch auto-harvests `/sys/fs/pstore` + `nvidia-bug-report.sh` into `athena-diag/crash-<ts>/`. efi-pstore survives the cold reboot; a pure silicon/PCIe hang (no NMI) is still uncapturable. **This works** — it caught the full `BUG`/`Call Trace`/GPF backtraces on 20260701-123219. (Caveat: `NVreg_EnableGpuFirmwareLogs=1` is a **confirmed no-op** — the `gsp_log_*.bin` decoder isn't in the public 595.71.05 firmware; the boot log shows `failed to load gsp_log_ga10x.bin`. Rely on `athena-kmsg-logger` + `nvidia-bug-report.sh`, not GSP firmware logs.)
- **`talk-llama` now fails SAFE on a GPU loss (brain-resilience, CHANGES.MD §10 L2).** ggml's CUDA backend has no soft error path — any CUDA error → `GGML_ABORT`/`abort()` (uncatchable), which is how the brain died at `state_write`. It now calls `gpu_context_healthy()` (a `dlsym`'d `cudaPeekAtLastError`/`DeviceSynchronize`, no CUDA link dep) before the first-reachable GPU ops (STT `whisper_full`, snapshot take/restore, `decode_tokens`, response decode) and, on a poisoned context, **ends the session cleanly with `_exit(0)`** — skipping GPU consolidation/frees (which would re-abort) but still writing the CPU-side session timestamp. It cannot keep serving (single-GPU MPS requires the client to exit), but a peer GPU fault no longer causes a SIGABRT, ERR-trap, or lost memory timestamp. A clean `_exit(0)` here means the launcher sees a normal shutdown, not a crash.
- **`nvidia-smi` clock/power tuning is a NO-OP in this container — don't rely on it (CHANGES.MD §12 W4).** The boot NVRM log shows the SBIOS power handshake is dead (`PlatformRequestHandler failed to get target temp / platform power mode from SBIOS`), so `nvidia-powerd` owns the ceiling and every *set* is discarded: `nvidia-smi -pl` is **rejected** ("not supported in current scope"; the 20260701-123219 run requested 85 W, readback stayed 105.35 W) and the `-lmc` memclk floor **doesn't hold** (9001 MHz at idle despite a `[13000,max]` lock). So `setup_gpu_performance` now **issues no `-pl` by default** (`ATHENA_GPU_POWER_LIMIT`/`ATHENA_GPU_POWER_MAX=1` opt-in only) and defaults **`ATHENA_LOCK_MEMCLK=0`** (`=1`/`=<floor>` opt-in). Crucially, 123219 rode the *same* ~95 W wall as the 074343 crash yet threw **no Xid** — stability comes from the **S0ix-off / RTD3-off D0-pin (§11 F1), not clock/power tuning**. `ATHENA_GPU_CLOCK_CAP` (`-lgc`) is likely dropped too and would cost throughput if honored — escalation-only.
- **BOTH GPU daemons auto-restart on userspace death (watchdogs, CHANGES.MD §12 W1 + §20.4).** A GPF/SIGSEGV/OOM in `llama-server` OR `orpheus-speak` is a *contained* failure — MPS sees a normal client exit, the GPU stays healthy, the brain keeps running. `llama_watchdog` polls a `/dev/shm` pidfile (`kill -0` fast path) + `curl :8080/health`, **plus a `LLAMA_WEDGE` flag from the keepalive deep probe** (§20.5: `/health` is a hardcoded "ok" on an independent HTTP thread pool — it can NOT see a decode wedged inside a CUDA call; the idle-gated keepalive's real 1-token completion reports instead, 3 fails with health green → force-restart). `orpheus_watchdog` (§20.4) is `kill -0`-only and pairs with the binary's **startup-recovery** (§20.3): a respawned orpheus-speak consumes a stale `---END---` trigger and writes `.done COMPLETE`, releasing a brain blocked on the handshake — without that, a completed trigger never fires inotify again and a respawn alone cannot unblock the brain. Both share the stop sentinel, rolling caps, and `<run>/watchdog.log`; `cleanup()` re-syncs both PIDs from pidfiles. Neither can save a device Xid (poisons all MPS clients). `ATHENA_WATCHDOG=0` disables both.
- **Never freeze the brain on a dead peer (CHANGES.MD §20.1/20.2 — the 134528 hang class).** The `.done` wait deadline is frozen at loop entry; a barge detected MID-loop must **rescale** it (`max_polls = min(max_polls, i+100)`) or a dead daemon means a silent 30-minute poll loop with all output gated off ("Athena went down"). The goodbye-path `stream_tts_wait_done` is bounded at 120 s (one-shot utterances only). If you add ANY new cross-process wait: bound it, rescale it on state changes inside the loop, and print on timeout.
- **Core dumps default OFF (`ATHENA_COREDUMP=0`, CHANGES.MD §19a).** They were scoped to the small GPU clients (llama-server + orpheus-speak `ulimit -c unlimited`; talk-llama forced `-c 0` — its ~161 GiB mlock'd RSS must never core), but on the corrupting box the coredump's large ext4 write draws poisoned pages: the 20260702-124334 fatal oops was INSIDE `do_coredump→…→folio_alloc`, turning a survivable client abort into a reboot. Re-enable with `ATHENA_COREDUMP=1` only on a non-corrupting kernel; then next libcuda GPF → `gdb <bin> <core> -ex bt`.
- **"Bad page state / pagealloc: memory corruption" is a KERNEL use-after-free, triggered by nvidia-smi UVM churn — NOT an ATHENA or GPU-fault bug (CHANGES.MD §13).** Root cause (research-backed): a Linux memcg/huge-vmalloc `split_page`/`page->private` UAF (CVE-2026-43303 class; same signature repros with OpenZFS, so it's kernel-side, driver-agnostic), *triggered* by NVIDIA UVM's large `kvmalloc→vmalloc` allocations. On this build **every `nvidia-smi` opens `/dev/nvidia-uvm` and does `UVM_INITIALIZE`** — so the monitor's per-sample nvidia-smi spawns (~3/sec) churned ~thousands of UVM va-space create/destroy cycles and surfaced it. `NVreg_PreserveVideoMemoryAllocations=0` did **not** fix it (§12 W5 refuted). §13a fixes the trigger (persistent `nvidia-smi --loop-ms` in the monitor → ~500× fewer cycles); §13b moves orpheus-speak's player to `posix_spawn` (out of the fork/`dup_mmap` victim path). **`page_poison=1` is the only thing CONTAINING it** — non-fatal *because* each poisoned page is discarded; without it the UAF is silent and uncontained (the likely substrate under the 074343 panic / 123219 GPF). **FINAL MODEL (§16, six crashes cross-tabulated, adversarially verified): in-place SINGLE-BYTE corruption of live kernel page metadata by CPU-side kernel code — the one disease behind every panic.** 4 of 5 terminal oopses GPF at `lruvec_stat_mod_folio` on a garbage folio→memcg pointer (instruction-decode-proven; one byte scribbled in an otherwise-valid pointer); the 5th is a corrupted page-cache XArray. **Falsified:** huge-vmalloc as sole variant (crashed WITH `nohugevmalloc`); alloc-time hygiene as sufficient; **errant device DMA (the dGPU sits in a translated `DMA-FQ` IOMMU domain by Debian default — `INTEL_IOMMU_DEFAULT_ON_INTGPU_OFF=y` — with zero DMAR faults; the corrupted regions are never IOVA-mapped).** **Xid 69 is a co-symptom** (2/5 fatal runs had none); the correlate is UVM/driver activity, not the power wall. Leading hypothesis: CVE-2026-43303-class stale-struct-page bug exercised by driver/UVM kernel paths. **The fix is a kernel: 6.12.94-1 (trixie-security, already in /boot — bootable via GRUB Advanced options) or ≥7.0.13-1** (`check-kernel-fix.sh` watches backports). Exposure shrinkers meanwhile: `athena-uvm-hardening.conf` (`uvm_disable_hmm=1 uvm_ats_mode=0`, zero cost, installed by `apply-gsp-stability.sh`) + driver 595.84. Keep `page_poison=1 nohugevmalloc` as tripwires until a fixed kernel + clean soaks. Do NOT: add `intel_iommu=on` (no-op), build Xid-triggered defenses, or chase pinned-memory removal (with `GGML_CUDA_NO_PINNED=1` — now defaulted in the launcher — zero app-level pinned registrations remain).
- **The dialogue antiprompt (user turn tag) must stay NEWLINE-ANCHORED (CHANGES.MD §14).** talk-llama stops generation on the user tag; as bare `"Igor:"` it matched MID-SENTENCE when the model addressed the user by name + colon ("…This is peak Igor: …", run 20260701-163723) — truncating the reply, leaving a user tag on the context tail, and causing a verbatim replay loop on every re-ask (copy attractor; `--repeat-penalty 1.0` is disabled by design). `antiprompts[0]` is now `"\n" + person + chat_symb` (matching the always-anchored bot tag and how every real turn boundary is written), with a tail-only bare-tag strip for the flushed-newline case. Don't "fix" a truncation by un-anchoring; if a model legitimately emits an unanchored tag, that's a prompt-format bug, not a matcher bug.
